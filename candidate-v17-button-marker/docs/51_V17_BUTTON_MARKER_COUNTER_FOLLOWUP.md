# OpenPatch RE v17 Follow-up: Stable Per-Button Marker/Counter Metadata

## Role and scope

This is a narrow follow-up to the OpenPatch RE v17 candidate. It keeps all original v17 requirements active and adds stable per-button marker/counter metadata so Codex and other agents do not rely on whole-page button counts while ChatGPT lazily loads or unloads older messages.

## Requirement addressed

Codex must be able to inspect a loaded message and identify the exact OpenPatch RE button context without needing full page history.

Added DOM data attributes on each `.openpatch-inline-button`:

- `data-openpatch-re-button-id`
- `data-openpatch-re-button-index`
- `data-openpatch-re-visible-count`
- `data-openpatch-re-message-id`
- `data-openpatch-re-message-button-index`
- `data-openpatch-re-message-visible-count`
- `data-openpatch-re-counter-scope`
- `data-openpatch-re-asset-kind`
- `data-openpatch-re-route-decision`
- `data-openpatch-re-allowed-action`
- `data-openpatch-re-not-approval`

The exact requested fields are present, and additional message-local fields are included because global visible counts can change under lazy loading.

## Semantics

- `button_id`: per-button UUID generated by the content script.
- `button_index`: 1-based index among currently loaded visible OpenPatch RE buttons.
- `visible_count`: number of currently loaded visible OpenPatch RE buttons.
- `message_id`: compact hash derived from the closest ChatGPT message container, explicit message id/test id when available, or compact text/link/button hints.
- `message_button_index`: 1-based index among OpenPatch RE buttons inside the same loaded message.
- `message_visible_count`: number of OpenPatch RE buttons inside the same loaded message.
- `counter_scope`: `loaded_dom_and_current_message`.
- `asset_kind`, `route_decision`, `allowed_action`, `not_approval`: copied into both marker JSON and DOM data attributes for low-cost agent decisions.

## Lazy loading behavior

The global `button_index` and `visible_count` intentionally describe the loaded DOM only. They are not treated as historical page totals.

The stable local identity is:

```text
pageSessionId + message_id + message_button_index + button_id
```

This lets Codex inspect the current loaded message and choose the correct button without requiring unloaded history.

- It does not claim real GitHub upload/apply success.
- It does not install the live Auto Continue adapter.
